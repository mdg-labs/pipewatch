---
title: Permissions and events
description: Required GitHub App permissions and webhook events for PipeWatch to ingest workflow runs and jobs.
---

PipeWatch reads GitHub Actions metadata via webhooks and the REST API. Request only the permissions below — excess scopes slow org security review and are unnecessary for MVP.

## Repository permissions

| Permission | Access | Purpose |
|---|---|---|
| **Actions** | Read | List workflow runs, jobs, and steps |
| **Metadata** | Read | Repository names, default branch, visibility |
| **Contents** | Read (optional) | Workflow file paths for display labels |

**Actions** and **Metadata** (read) are required. **Contents** is optional — enable it if you want workflow file paths shown in the UI.

## Organization permissions

No elevated organization permissions are required for MVP. Install the app on each org or user account you want to monitor.

## Subscribe to events

Enable these webhook events on the GitHub App under **Permissions & events → Subscribe to events**:

| Event | Purpose |
|---|---|
| `workflow_run` | Run queued (`requested`), in progress, completed |
| `workflow_job` | Job queued, in progress, waiting, completed |
| `installation` | App installed, suspended, or removed |
| `installation_repositories` | Repos added or removed from an installation |

PipeWatch validates every payload with `X-Hub-Signature-256` (HMAC-SHA256) before processing. Invalid signatures return **401** with no side effects — there is no bypass flag.

## What PipeWatch does not need

These are **out of scope** for MVP — do not grant write access:

| Permission | Why not |
|---|---|
| Actions (write) | PipeWatch does not trigger or re-run workflows |
| Contents (write) | No code changes |
| Administration | No repo settings changes |
| Pull requests, Issues, etc. | Not used for pipeline visibility |

## Polling fallback

When webhooks cannot reach your API, enable **polling** per repository in PipeWatch settings or set `PIPEWATCH_MODE=polling` globally for CE.

Polling uses the GitHub REST API with the installation token — the same **Actions (read)** permission covers both modes. See [Webhook vs polling mode](/docs/concepts/webhook-vs-polling) for trade-offs and intervals.

## Verify configuration

After saving permissions:

1. Install or re-install the app on a test repository.
2. Trigger a workflow run on that repository.
3. In GitHub → App → **Advanced**, confirm `workflow_run` deliveries succeed (HTTP 200).
4. Check the PipeWatch dashboard for the new run within seconds (webhook mode).

If deliveries fail, see [Webhook URL](/docs/github-app-setup/webhook-url) troubleshooting or set up a [Cloudflare Tunnel](/docs/github-app-setup/cloudflare-tunnel).
